## Goal
Define and implement the validation boundary that determines whether model output can be persisted, handed off, or rejected.

## Acceptance Criteria
- [ ] Validation rules are consistent with the guardrail and commit policy.
- [ ] The write boundary makes allowed repository mutations explicit.
- [ ] Rejected outputs produce actionable diagnostics for retry or review handoff.
- [ ] The validation boundary is separable from provider execution for independent testing.

## Notes
- High-risk shared hotspot; do not spread this logic across unrelated tickets.
- This ticket is the main safeguard against repository damage from bad model output.
