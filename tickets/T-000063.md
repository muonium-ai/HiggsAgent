## Goal
Create directories, files, and initial project scaffolds from normalized autonomous coding-session outputs.

## Acceptance Criteria
- [x] Materialize a normalized autonomous response into directory creation operations for nested folder structures.
- [x] Materialize new files from structured code or text payloads without relying on manual operator copy-paste.
- [x] Record created paths and artifact metadata so the existing write gate and observability pipeline can inspect them.
- [x] Reject ambiguous or malformed scaffold instructions with actionable error output instead of partial silent writes.

## Notes
- This ticket covers the capability gap shown by scaffold-style model responses that describe `mkdir`, `touch`, and new file contents.
- Runtime support now accepts normalized `scaffold` payloads, nested scaffold trees, and `text` aliases for file content while rejecting duplicate materialization paths before writes begin.
- Execution artifacts now persist `materialization-plan.json` so review-mode bundles and observability consumers can inspect the intended directory and file writes.
